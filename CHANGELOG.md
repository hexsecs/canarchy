## [Unreleased]

### Added

* **`canarchy j1587` — J1587/J1708 legacy heavy-vehicle decoding (#415).** A new `canarchy j1587` command group brings legacy heavy-vehicle diagnostic captures to the structured-output surface, mirroring `canarchy j1939`. `j1587 decode --file <path>` parses a J1708 capture file (lines of `(timestamp) j1708 <hex>`, each the source MID plus PID-framed parameters and a checksum byte) into one `j1587_parameter` event per parameter, validating the J1708 byte-sum checksum and resolving common PIDs to names/units/values via a bundled catalog (`resources/j1587/pids.json`, override via `CANARCHY_J1587_PID_OVERRIDES`); `j1587 pids` lists that catalog as a reference payload. The parser is a clean-room implementation of the SAE J1587 PID framing rules (1/2/explicit-length/extended-PID parameters) and does not reuse code or data from bundled third-party tools. `--offset`/`--max-frames`/`--seconds` bound analysis like `j1939 decode`; a missing file returns `J1587_SOURCE_UNAVAILABLE` and a malformed line returns `J1587_SOURCE_INVALID` (exit 1). New `J1587ObservationEvent` / `j1587_parameter` canonical event, `j1587_decode` / `j1587_pids` MCP tools, a J1708 fixture, and design/test specs (`j1587-decode.md`). Closes #415.

* **`fuzz guided` — response-feedback coverage-guided fuzzing (#350).** AFL-style instrumentation coverage is unavailable when the target is an ECU across a bus, so `canarchy fuzz guided <interface> --id <arb-id>` derives its novelty signal from the target's *observed responses*. A new response-fingerprint engine (`src/canarchy/fuzz_feedback.py`) reduces each response to behaviour markers — UDS negative response codes (`nrc:<svc>:<code>`), UDS positive responses, DM1 fault emergence (`dm1:<spn>:<fmi>`), response-timing buckets, and silence transitions — selectable via `--signals`. A guided loop (`src/canarchy/fuzz_guided.py`) on top of the `canarchy.fuzzing` havoc/splice mutators keeps inputs that elicit previously-unseen markers as corpus seeds and prioritises their lineage; the corpus is capped (`--max-corpus`, lowest-scoring pruned first) and persisted to `--corpus <dir>` (raw seed files plus a `lineage.json` manifest) so campaigns resume. Campaigns are bounded by `--max-iterations` / `--max-seconds`, the envelope reports `iterations` / `new_behaviour_count` / `corpus_size` / `unique_markers` / `stop_reason` / `findings`, and the loop is deterministic under a fixed `--seed`. It honours the active-transmit safety model (`--ack-active`, `--rate` pacing, `--dry-run` plans without transmitting); structured errors `FUZZ_GUIDED_INVALID_SIGNALS` / `FUZZ_GUIDED_INVALID_ID` (exit 1) and `FUZZ_GUIDED_TRANSPORT_FAILED` (exit 2). The `fuzz_guided` MCP tool is gated like other active tools (mandatory `ack_active=true`, `dry_run` defaulting to true). Each iteration uses a new `LocalTransport.transaction` send+receive primitive (one bus, receive path active before the probe is sent) so a fast response is not missed and recorded as false silence; `uds scan` and `xcp scan` now use the same primitive. Tests use in-process mocked responders (no live bus) covering fingerprinting, scoring, determinism, lineage pruning, kill-switch, corpus persistence, the CLI dry-run/active paths, and the MCP gate. New design/test specs (`response-feedback-fuzz.md`). Closes #350.

* **`re suggest` — heuristic + optional-LLM signal-name suggestions (#332).** After `re signals` ranks changing bit-fields, `canarchy re suggest <file>` proposes names for each candidate. The default path is fully offline and combines four heuristic sources — overlap with a reference DBC's signals (`--reference-dbc`, a path or provider ref), bit-range overlap with the bundled J1939 SPN catalog, the PGN name as a coarse fallback, and a plain-English template from the candidate's change behaviour — each suggestion carrying a `source` (`dbc`/`spn`/`pgn`/`heuristic`/`llm`) and `confidence`, with the top reported as `suggested_name`/`suggested_source`. An optional, off-by-default `--llm <provider>` flag (only `anthropic`) enriches names via an external LLM behind an explicit confirmation gate (`--yes`, a `YES` reply, or `CANARCHY_LLM_NONINTERACTIVE=1`); it sends only candidate metadata — never raw payload bytes — and records an `external_enrichment` note plus an `EXTERNAL_SERVICE_CALLED` warning. Declining returns `LLM_CONFIRMATION_DECLINED`; an unknown provider returns `LLM_PROVIDER_UNSUPPORTED`; a missing key returns `LLM_PROVIDER_UNAVAILABLE`. The `re_suggest` MCP tool exposes the heuristic path only — the LLM enrichment is CLI-only. New modules `src/canarchy/re_suggest.py` (pure heuristics) and `src/canarchy/llm_suggest.py` (feature-flagged, injectable provider client); tests cover the heuristic path against an EEC1 fixture with the LLM path mocked. Privacy/usage notes added to `SECURITY.md` and `docs/agents.md`; new design/test specs (`re-suggest.md`). Closes #332.

* **XCP (measurement/calibration) workflows for XCP-on-CAN (#327).** A new `canarchy xcp` command group brings ASAM MCD-1 XCP coverage to the structured-output surface: `xcp scan <interface>` transmits an XCP CONNECT and reports each responder (parsing the CONNECT response into resources, max-CTO/max-DTO, and protocol/transport versions); `xcp trace <interface>` pairs command/response CTOs into named transactions with positive/error status (error codes resolve to ASAM names); `xcp read <interface>` surfaces raw DAQ measurement (DTO) payloads; and `xcp commands` lists the bundled command catalog. The implementation is a pure parser in `src/canarchy/xcp.py` wired through the standard transport layer (the scaffold backend returns deterministic sample data, so tests need no live hardware), emitting new `xcp_transaction` / `xcp_measurement` canonical events. `xcp scan` is active and honours the active-transmit safety model (`--ack-active`, `--dry-run` plans the CONNECT frame without transmitting); request/response CAN ids default to 0x3E0/0x3E1 and are overridable via `--request-id` / `--response-id` (a malformed id returns `XCP_INVALID_ID`; an extended 29-bit request id is sent as an extended frame). All four subcommands are exposed as MCP tools (`xcp_scan`, `xcp_trace`, `xcp_read`, `xcp_commands`), with the active `xcp_scan` tool gated like other active-transmit tools (mandatory `ack_active=true`, `dry_run` defaulting to true). Signal-level decoding of DAQ payloads (which needs the slave's A2L) and XCP-on-Ethernet/USB transports are out of scope for v1. New design/test specs (`xcp-workflows.md`). Closes #327.

* **DoIP (Diagnostic over IP) transport for UDS workflows (#326).** `uds scan` and `uds trace` now accept a `doip://<host>:<port>?logical_address=0x0E80` target and run UDS discovery/tracing over a TCP/IP gateway (ISO 13400-2), reaching an ECU by 16-bit logical address instead of a CAN arbitration id. A new pure codec (`src/canarchy/doip.py`) encodes/decodes the DoIP generic header, routing-activation, and diagnostic-message payloads; the connection performs routing activation and a diagnostic-message exchange, then emits the same canonical `uds_transaction` envelope (with `transport: doip`, `host`/`port`/`logical_address`/`source_address`) so existing consumers and output formats are unchanged. `uds scan` enumerates the default/programming/extended sessions; `uds trace` runs a DiagnosticSessionControl + TesterPresent exchange. Both are active TCP egress and are gated by the active-transmit safety model (`--ack-active`, `[safety].require_active_ack`, `YES` confirmation), and UDS negative responses surface as transactions rather than errors. Optional target query parameters: `source_address` (default `0x0E00`), `activation_type` (default `0x00`), `timeout` (default `2.0s`); port defaults to `13400`. Structured errors: `DOIP_INVALID_TARGET` (exit 1), `DOIP_CONNECTION_FAILED` / `DOIP_TIMEOUT` / `DOIP_ROUTING_ACTIVATION_DENIED` / `DOIP_DIAGNOSTIC_NACK` / `DOIP_PROTOCOL_ERROR` (exit 2). DoIP is a CLI-only operator action: the MCP `uds_scan` / `uds_trace` tools refuse a `doip://` target with `DOIP_MCP_EXCLUDED` (active network egress to an arbitrary host, like `cannelloni send`). Tests use an in-process loopback DoIP responder (no live network) covering codec round-trips, session-enumeration scan, trace, silent-probe skip, invalid-target, unreachable-endpoint, diagnostic-NACK, the active-transmit gate, and the MCP rejection. New design/test specs (`doip-diagnostic-workflows.md`). Closes #326.

* **cannelloni CAN-over-UDP interop (#418).** CANarchy now speaks the [cannelloni](https://github.com/mguentner/cannelloni) UDP datagram wire format (version 2), so captures can be exchanged with cannelloni endpoints such as the UTHP / TCAT heavy-vehicle appliances' remote-bus setups. A pure codec (`src/canarchy/cannelloni.py`) round-trips classic CAN, extended, RTR, error, and CAN FD frames (BRS/ESI), carrying the SocketCAN EFF/RTR/ERR flag bits in the can_id. Two commands: `canarchy cannelloni decode --file <payload>` decodes one or more concatenated datagrams into canonical `frame` events (passive, file-backed, exposed as the `cannelloni_decode` MCP tool); `canarchy cannelloni send <host:port> --file <capture>` transmits a capture as UDP datagrams under the active-transmit safety model (`--ack-active`, `--dry-run` plans without a socket, `--max-count`/`--rate`/`--seq-no` controls), CLI-only as it is active UDP egress to an arbitrary host. Documented as a transport option in `docs/backends.md`, with design/test specs (`cannelloni-interop.md`). Tests cover a hand-computed wire reference, all-frame-type round-trips, truncation/version errors, the decode CLI, send `--dry-run` planning, and a loopback-UDP send round-trip. Closes #418.

* **`canarchy web serve` — read-only browser dashboard over the JSONL envelope (#324).** A new presentation tier streams capture files to a bundled single-file SPA: live frame stream, bus status (frame rate, unique ids), decoded signals when a `--dbc` database is supplied, J1939 PGN/source-address activity with bundled labels and SA names, and recent UDS transactions reassembled from the capture. The server is dependency-light by design — standard-library HTTP plus a minimal RFC 6455 WebSocket implementation, and a vanilla-JS SPA with no frontend framework — and strictly read-only: no active-transmit endpoints exist, and every non-GET request returns 405 with a structured `WEB_READ_ONLY` error. Timestamp-paced streaming honors `--rate` (gaps capped at 1 s) and `--loop`; `--bind` defaults to `127.0.0.1:8474` with ephemeral-port support; the startup envelope reports the resolved `url`, `read_only: true`, and event counts; bounded-analysis flags (`--offset`/`--max-frames`/`--seconds`) apply. `web serve` is documented as an MCP exclusion (long-running front end, like `shell`/`tui`). Tests cover event construction (frame/J1939/decoded/UDS, timestamp ordering), server startup against a fixture capture, a raw-socket WebSocket smoke check (handshake accept key, envelope events, `STREAM_COMPLETE` close), the read-only 405 surface, bind validation, WS framing round-trips, and the CLI envelope. New design/test specs (`web-serve.md`) and a tutorial (`docs/tutorials/web_dashboard.md`). Closes #324.

* **decode→encode round-trip: `encode` resolves SAE display names (#413).** Signals you can decode can now be re-encoded by the same name: `encode EEC1 "Engine Speed=1200"` works against a DBC that defines `EngineSpeed` inside `EngineSpeed1`. Message names resolve by exact DBC name, case/spacing-insensitive match, or SAE PGN label/name from the bundled J1939 catalog (PGN-label ties across source addresses are broken by the supplied signal names; remaining ambiguity returns `DBC_MESSAGE_NOT_FOUND` listing the candidates). Signal names resolve by exact DBC name, case/spacing-insensitive match, or the bundled SAE SPN name of an SPN-attributed signal. Unsupplied signals are defaulted (DBC initial value, else 0 clamped into range/choices; multiplexed messages excluded) so single-signal encodes succeed. Every non-exact resolution and defaulted signal is reported under `data.resolution` and as warnings, misspelled names get closest-match suggestions in the error hint, and `send --dbc` applies the same behavior with a transmission-specific warning. Tests cover the issue's exact example, a byte-identical decode→encode round-trip, SPN-name resolution, the PGN tie-break, defaults reporting, and suggestion hints. Closes #413.

### Fixed

* **`j1939 summary` dedupes repeated BAM identifiers (#411).** The same printable TP identifier (e.g. a VIN broadcast once per BAM session) was listed once per session, inflating the response 14× on real drive captures. `tp.printable_identifiers` now reports distinct values with an `occurrence_count` and the source-address name, matching the deduped shape `j1939 compare` already used; `j1939 compare`'s passthrough inherits the fix. Closes #411.
* **`filter` accepts decimal IDs/PGNs and no longer returns a misleading empty data block on errors (#414).** `id==`/`pgn==` operands now accept decimal, `0x`-prefixed hex, or bare hex, and all operators (`==`, `>`, `~=`) tolerate surrounding whitespace — `filter "pgn == 61444"` now works instead of returning `INVALID_FILTER_EXPRESSION`. On an invalid expression the JSON error envelope no longer carries `frame_count: 0` / `frames: []`, so a caller checking only `frame_count` can no longer mistake a bad query for a successful zero-match. The error hint documents the accepted operand forms. Closes #414.

* **MCP responses are now bounded and per-tool failures are isolated (#405).** A single oversized tool result (e.g. `j1939_pgn` on a high-rate PGN emitting ~22 MB of per-frame JSON) could disconnect the stdio MCP server and leave every tool unavailable for the rest of the session. Every MCP response is now capped (default 512 kB, configurable via `CANARCHY_MCP_MAX_RESPONSE_BYTES`): oversized list-shaped data is trimmed with an explicit `data.truncated: true` marker and a `data.truncation` block recording each trimmed list's original `total_items` vs `returned_items` plus a CLI re-run hint, so an agent can never mistake a truncated result for a complete zero-match; data that cannot be reduced by list trimming falls back to a stub that preserves the envelope. Unexpected in-tool exceptions now return a canonical `TOOL_EXECUTION_ERROR` envelope instead of propagating into the transport, so one failing call leaves the rest of the session usable. Added `REQ-MCP-20`/`REQ-MCP-21` and tests (`TEST-MCP-39`–`43`) covering a 4000-frame high-rate EEC1 capture against `j1939_pgn`, list truncation totals, the giant-scalar fallback, and post-exception session usability. Closes #405.
* **RE heuristics are now J1939-transport-aware (#407).** `re counters` and `re signals` no longer report J1939 TP framing as discoveries — TP.DT sequence numbers previously surfaced as near-perfect "counters". Transport-protocol PGNs (TP.CM 60416, TP.DT 60160, ETP.CM 51200, ETP.DT 50944) are excluded from counter/signal detection and reported under `excluded_transport_ids` (with a warning noting the exclusion), while `re entropy` / `re anomalies` / `re corpus` label such ids with `j1939_transport: true` and a rationale note instead of presenting them as application signals. Closes #407.
* **No-baseline `re anomalies` no longer ranks sparse ids on artifact z-scores (#408).** Self-consistency mode previously flagged low-rate/bursty ids with astronomical z-scores (e.g. 156,742σ) produced by a near-zero spread estimate, drowning out real anomalies. Without a baseline, an id now needs at least `--min-samples` inter-frame gaps (default 10; 3 with a baseline) before its timing is scored — sparser cyclic-looking ids are reported under `low_rate_ids` with classification source `low-sample` instead of being ranked. Reported z-scores are capped at ±100σ (`z_score_capped: true`, rationale says "capped"), and no-baseline runs emit a warning nudging the operator toward `--baseline`. The `min_samples` knob is exposed on the CLI and the `re_anomalies` MCP tool. Closes #408.

### Added

* **Enriched `canarchy stats` with per-ID frequency, timing, DLC spread, and a bus-load estimate (#410).** `stats` previously returned strictly less than `capture-info` (total frames + unique-ID count only). It now also reports capture duration and first/last timestamps, a `dlc_distribution`, a `bus_load` block (total bits, bits/s, and load percentages at 250 k / 500 k / 1 M bit/s — frame overhead without stuff bits, documented as a lower-bound estimate), and a bounded `top_ids` per-ID breakdown (default 20, configurable via `--top`) carrying `arbitration_id_hex`, frame count, share of traffic, rate in Hz, mean/min/max inter-frame gap, gap jitter, observed DLCs, and first/last seen. Both the file-backed and stdin paths gain the new fields; existing fields are unchanged. Closes #410.
* **Bundled SAE J1939-73 FMI catalog and an OEM SPN extension path (#409).** DM1 DTCs and `j1939 faults` entries now carry `fmi_description` resolved from a new bundled FMI table (`canarchy/resources/j1939/fmis.json`), turning numeric failure modes into readable diagnostics out of the box (the bundled SPN catalog already resolves ~16 k standard SPN names). For OEM/proprietary SPN ranges there is now a documented extension path: a JSON file shaped like the bundled `spns.json` (`{"<spn>": {"name": ...}}`) resolved from `$CANARCHY_J1939_SPN_OVERRIDES` (or `~/.canarchy/j1939_spns.json`) is merged over the built-in catalog, and a `--dbc` still overrides per matched SPN. Closes #409.
* **J1939 PGN / source-address annotation on all RE-tool results (#406).** `re signals`, `re counters`, `re entropy`, `re anomalies`, and `re corpus` candidates now carry `pgn`, `pgn_label`, `pgn_name`, `source_address`, and `source_address_name` (resolved from the bundled J1939 metadata) whenever the frame is a 29-bit J1939 id, plus `arbitration_id_hex` on every candidate — so `re entropy` now says "0x18FEBF0B = PGN 65215 (EBC2) from SA 11 (Brakes - System Controller)" instead of a bare decimal id. Annotation is omitted gracefully for non-J1939 (11-bit) ids. Closes #406.
* Added `canarchy simulate [<interface>] --profile {heavy-truck,passenger-car} [--rate <hz>] [--duration <seconds>] [--seed <n>]` (#333) — a standalone, data-driven bus simulator that emits a deterministic, seedable mix of classic CAN frames, J1939 PGN traffic, and occasional DM1 fault bursts shaped by a named vehicle archetype. Profiles live as plain JSON resources under `canarchy/resources/simulate/profiles.json` (`heavy-truck`: instrument-cluster/body-controller heartbeats plus EEC1/ET1/CCVS/TC1/EBC1 J1939 traffic and an SPN 110 coolant-temperature DM1 burst; `passenger-car`: dense proprietary CAN plus a thinner J1939 slice and an SPN 84 vehicle-speed DM1 burst), so new archetypes can be added without touching `canarchy.simulate`. Frame templates are sampled with a seeded `random.Random` weighted by each template's `weight`, timestamps are spaced at `1 / --rate`, and J1939/DM1 arbitration IDs and payloads are built with the existing `compose_arbitration_id` and a DM1 byte layout compatible with `canarchy.j1939.dm1_messages` decoding (round-trips cleanly). `simulate` is wired into the existing active-transmit safety gate (`--ack-active` / `YES` confirmation / `[safety].require_active_ack`), supports `--dry-run` planning, and reuses the existing transport backends — including `socketcan` virtual interfaces, `udp_multicast`, and stdout candump piping — and candump text rendering, so no new transmission or output machinery was needed. The `simulate` MCP tool mirrors the CLI with a mandatory `ack_active` and `dry_run` defaulting to `true`. Added 8 CLI tests in `tests/test_cli.py` (dry-run planning, deterministic seeding, frame-mix verification against the scaffold backend, active transmission, text rendering, and structured `SIMULATE_INVALID_RATE` / `SIMULATE_INVALID_DURATION` / argument-parser validation errors) and MCP coverage in `tests/test_mcp.py` (ack/dry-run gating and `_build_argv` translation). Added design spec `docs/design/simulate-command.md`, test spec `docs/tests/simulate-command.md`, a cookbook recipe (`docs/cookbook/simulate-vehicle-profiles.md`) covering the profile JSON schema, and a tutorial (`docs/tutorials/simulate_vehicle_traffic.md`) walking through planning, transmitting, and analyzing simulated traffic. Closes #333.
* Added `canarchy plugins list|info|enable|disable` (#317), exposing the existing Python entry-point plugin registry through the canonical CLI envelope. Operators can list discovered processor, sink, and input plugins with name, kind, API version, package version, source distribution, enabled state, and configured options; `plugins info <name>` shows matching metadata and config for a single plugin; enable/disable persists `[plugins."<name>"].enabled` in `~/.canarchy/config.toml`. Read-only MCP mirrors `plugins_list` and `plugins_info` were added, while toggles remain CLI-only because they write user config. Updated plugin specs, command docs, agent docs, MCP coverage, troubleshooting, and the feature matrix. Closes #317.
* Added `canarchy re anomalies <file>` — passive timing/ID anomaly detection (#321). Flags inter-frame-timing outliers and, against a `--baseline` capture, unexpected (`unknown-id`) and missing (`dropped-id`) arbitration IDs; each anomaly carries arbitration id, kind, score, z-score, sample count, timestamp, and a rationale, ranked by score. Without a baseline it scores each ID against its own statistics (self-consistency). Crucially, **only IDs judged cyclic are timing-checked, so event-based and event-periodic messages are not falsely flagged**: classification is authoritative from a `--dbc` (a message's `cycle_time` / send type decides cyclic vs event) and otherwise from a robust coefficient of variation against `--cv-max` (default 0.5). Timing statistics use the median and median-absolute-deviation rather than mean/stdev, so a minority of outlier gaps cannot inflate the spread and mask themselves. Honors the standard `--offset` / `--max-frames` / `--seconds` window flags, the canonical `--json` / `--jsonl` / `--text` envelope, and is mirrored by the `re_anomalies` MCP tool. The detector lives in `canarchy.reverse_engineering.anomaly_candidates` with a `database_timing_map` helper in `canarchy.dbc`. Added anomaly fixtures and unit + CLI + MCP tests in `tests/test_reverse_engineering.py` and `tests/test_mcp.py`. Updated `docs/command_spec.md`, the RE design/test specs, `docs/agents.md`, and the `docs/feature-matrix.md` Reverse-engineering depth narrative. Closes #321.
* Added ARXML / KCD / SYM database read support (#320). The database-backed commands (`decode`, `encode`, `dbc inspect`, `dbc convert`, and the J1939 DBC-enrichment paths) now accept `.arxml`, `.kcd`, and `.sym` files in addition to `.dbc`, routed by filename suffix through the existing cantools runtime — no new parsers and the user-facing flag stays `--dbc`. A new `detect_database_format` helper drives provenance: `dbc inspect` reports the real loader in `data.database.format`, and every database command now includes `data.dbc_source.kind` (`dbc` / `arxml` / `kcd` / `sym`). Provider-backed resolution (`opendbc:<name>`) is unaffected. Added `tests/fixtures/sample.{arxml,kcd,sym}` and tests in `tests/test_cli.py` (format/kind provenance, DBC↔KCD encode and decode parity) and `tests/test_dbc_provider.py` (ARXML/KCD load integration plus the suffix detector). Updated `docs/command_spec.md`, `README.md`, and the `docs/feature-matrix.md` Database breadth row (CANarchy → Yes). Closes #320.
* Added `canarchy dbc convert <dbc> --to {dbc,kcd,sym} [--out <path>]` (#385), which serializes a loaded database into another cantools-supported format using the `as_dbc_string` / `as_kcd_string` / `as_sym_string` emitters (no hand-rolled writers). With `--out` the converted database is written to disk and the envelope reports `out`, `target_format`, `message_count`, and `signal_count`; without `--out` the serialized `content` is returned in the envelope and printed directly in `--text` mode. Structured errors cover unwritable output directories (`DBC_CONVERT_WRITE_FAILED`) and serialization failures where the target format cannot express a source feature (`DBC_CONVERT_FAILED`). The `dbc_convert` MCP tool mirrors the CLI (file generation is a developer action, so no active-transmit gate applies). Shell completion now lists the `convert` and `signals` subcommands and their flags. Tests cover a DBC→KCD round-trip with message/signal parity, stdout content, text-mode confirmation, the MCP argv mapping / schema, and the unwritable-output error path. Updated `docs/command_spec.md`, `docs/agents.md`, and the `docs/feature-matrix.md` Database breadth row (CANarchy → Partial; reading non-DBC formats is still tracked as #320). Closes #385.
* Added `canarchy dbc inspect --layout` (#384), which reuses cantools' bit-layout, signal-tree, and choice-table renderers. Text output now shows per-message diagrams, while JSON/JSONL include `layout`, `signal_tree`, and `signal_choices` fields for agent consumption.
* Added `canarchy dbc generate-c <dbc> [--out-dir] [--database-name] [--no-floating-point-numbers] [--bit-fields] [--use-float] [--node] [--use-round]` (#386), which generates C source/header/fuzzer files from a database using the cantools C source generator (`cantools.database.can.c_source.generate`). The output directory defaults to the current working directory and must already exist; `--database-name` sets the C identifier prefix (defaulting to the source filename stem). Four files are emitted per database: `.h` header with `pack`/`unpack` declarations, `.c` source, `_fuzzer.c` fuzzer source, and `_fuzzer.mk` fuzzer makefile. The envelope reports `out_dir`, `database_name`, `files` (path/kind/size per file), and `file_count`. Structured errors cover missing output directories (`DBC_GENERATE_C_DIR_MISSING`), write failures (`DBC_GENERATE_C_WRITE_FAILED`), and generation failures (`DBC_GENERATE_C_FAILED`). MCP is intentionally excluded — file generation is a developer action, not an agent tool call. New tests cover JSON/JSONL/text output, file existence, pack/unpack symbol presence, `--database-name` prefix, default-out-dir from cwd, missing-directory error, `--bit-fields`, and `--node`. Updated `docs/command_spec.md`, `docs/feature-matrix.md`, `docs/design/mcp-server.md` (exclusion matrix), `docs/agents.md`, and shell completion. Closes #386.
* Added PCAP/PCAPNG capture-file read support (#319). All file-backed CAN analysis commands (`capture-info`, `stats`, `filter`, `decode`, `j1939 decode`, `j1939 pgn`, `j1939 spn`, `j1939 tp`, `j1939 dm1`, `j1939 summary`, `j1939 inventory`, `j1939 compare`, `re signals`, `re counters`, `re entropy`, `re correlate`, `re anomalies`, and `re match-dbc`) now accept `.pcap` and `.pcapng` files with CAN SocketCAN (DLT 227) frames. A new `canarchy.pcap_reader` module uses `dpkt` (added as a core dependency) to parse pcap/pcapng containers and handles classic CAN, CAN FD, remote, and error frames with correct flag extraction. Supported on both pcap and pcapng formats via dpkt's `UniversalReader`. Added `tests/fixtures/sample.pcap` and `tests/fixtures/sample.pcapng` fixtures covering all frame types, 13 transport-level and 4 CLI-level tests, and updated `docs/command_spec.md`. Closes #319.

### Changed

* `plot --dbc` now accepts `provider:ref` shorthand (e.g. `opendbc:<name>`) like `decode`/`encode`, resolved through the provider registry (#427). The plot envelope now includes `dbc_source`, and a bad ref returns the standard `DBC_*` error (exit code 3) instead of a generic `PLOT_ERROR`. Local database paths are unchanged. The `plot` MCP tool schema and command spec are updated; closes the last DBC-consuming command that ignored provider refs (surfaced by the #425 docs sweep). Closes #427.

* `uds scan` and `uds trace` now reassemble ISO-TP multi-frame UDS responses, ignore flow-control frames, and mark partial transactions with `complete=false` when a segmented response is truncated or arrives out of order.

This prevents captures where every DM1 message contains only no-fault filler entries (e.g. SPN=255/FMI=0) from being reported as having hundreds of active faults.
* J1939 signal decoding now returns `null`/`None` for the `value` field when the raw signal data matches the J1939 not-available pattern (0xFF for 8-bit, 0xFFFF for 16-bit, 0xFFFFFFFF for 32-bit signals) instead of converting the error sentinel into a physically impossible reading.
